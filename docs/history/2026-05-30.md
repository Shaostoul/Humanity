# 2026-05-30 — Gameplay-loop arc (v0.329 → v0.334)

The session that turned the *wired* engine (the 2026-05-29 engine-wiring arc) into an
actually-**playable** production + survival loop. Operator vision (brain-dump): "develop as
if the user unlocked everything 100%… cook or eat raw with buffs/harm… a drone goes to a
finite asteroid for iron/nickel/platinum, returns, drops raw material to be processed into
usable material then components… what all do we need to cover holistically?" → I mapped it
in `docs/design/gameplay-loops.md` + proposed a build order; operator approved ("Sounds
good. Let's do that.").

## What shipped (each: passing tests + relay check + lint + exe archived + GitHub/forge)

- **v0.329.0 — #1+#2: GUI→ECS command bridge + real crafting.** The reusable backbone every
  later loop uses: a `GuiState` flag → the main loop writes a `DataStore` `Mutex` channel
  before `system_runner.tick` → the owning `System` drains it in `tick`. "Dev: stock all
  materials" provisions one stack of every recipe input (raws **and** intermediates — only
  raws left recipes needing an intermediate like `coal_0` uncraftable; `Inventory::ensure_slots`
  grows the bag). Craft button → real consume/produce on the ECS inventory.
- **v0.330.0 — #3: cooking + nutrition.** New `Vitals` (satiation/hydration) + `StatusEffects`
  components; `FoodSystem` registered + extended. Eat → restores from `food_system.ron`
  profiles; raw food rolls `raw_consumption_risk` → `food_poisoning`; a full meal → `well_fed`;
  hunger/thirst decay → `hungry`/`thirsty` conditions → starvation/dehydration health drain.
  `StatusEffectRegistry` (status_effects.csv) keeps durations/mods in data. Cooking = the
  existing crafting recipes (cook_soup…) → safer food.
- **v0.331.0 — #4: gardening.** Plant a seed (consume → spawn `CropInstance`), Water, Harvest
  (mature → produce per plants.csv yield → inventory → despawn). Garden panel + "Dev: grow all".
  Closes garden→cook→eat.
- **v0.332.0 — #5: drone↔asteroid mining** (the operator's core mechanic). `AsteroidBody`
  (finite multi-ore) + `Drone` (Outbound→Mining→Returning→Done) + registered `DroneSystem`.
  Commission a drone for an ore → it mines a finite asteroid → delivers ore home; an asteroid
  mined empty is deleted. Intent-collection avoids ECS borrow conflicts. Added nickel/platinum
  ore + test asteroids; Mining panel.
- **v0.333.0 — #6: refining-chain depth.** nickel/platinum/stainless ingots + smelt recipes;
  2-tier ore→ingot→alloy (stainless = iron + nickel ingots). Closes mine→refine→craft. Pure
  data + a test that every recipe input/output is a real item.
- **v0.334.0 — #3b: buffs are tangible.** `CameraController.speed_multiplier` (movement only,
  not look) set each frame from the player's active status-effect `speed` modifiers
  (`StatusEffectRegistry::net_stat_multiplier`). New `well_nourished` buff (+10%) on a good
  meal; thirsty/flu (−20%) now slow you. Delivers the operator's "increased move speed".

**Result:** the chain is playable end-to-end — **mine → refine → craft** and
**garden → harvest → cook → eat → buffs**. 9 ECS systems now tick (was 7).

## Key technical notes / lessons

- **The GUI→ECS bridge pattern** (flag → DataStore Mutex channel → System drain) scaled cleanly
  across 4 consumers (craft, eat, plant/water/harvest, commission-drone). It's the template.
- **hecs borrows:** `world.query(...).iter()` holds a shared column borrow; doing `world.get::<&mut T>`
  right after can conflict. Use `query_mut` (exclusive) for in-place mutation, and **intent-collection**
  (record entities/values during the query, mutate after it releases) when a tick touches multiple
  entities (mining mutates the drone + the asteroid + the player inventory).
- **Verify subagent output, don't trust it:** the read-only Explore agent that mapped the
  mining landscape gave a great map but its *code sketch* had real bugs (passing `&mut world`
  into a method while a `query_mut` held it; despawn-during-iteration; `get_mut` vs this hecs's
  `get::<&mut T>`). The shipped impl avoided all three.
- **Check the code before wiring:** the ECS `PlayerControllerSystem` no-ops today (no `PhysicsBody`
  player + no `physics_world` in the DataStore) — movement is camera-driven. So #3b's speed hook
  is the camera, not the dead system.
- **Windows-only LNK1318** ("PDB LIMIT (12)") still blocks linking the full `HumanityOS.exe` *test*
  bins locally (debug=0 doesn't help); `--lib` tests + `cargo check --features relay` + release
  builds are unaffected. engine_wiring_lint was verified by source inspection each time; CI/Linux
  runs the real binary.
- **CSV data quirk:** items.csv/recipes.csv are CRLF with a pre-existing mid-line `\r` in some
  rows — inserted new rows via an EOL-aware node splice (minimal-diff, not a full rejoin).
