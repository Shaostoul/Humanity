# 04-CONTRIBUTING

## Goal

Make it easy for humans and AIs to contribute safely with minimal onboarding.

## Before coding

1. Read [`00-START-HERE.md`](./00-START-HERE.md)
2. Read [`02-ARCHITECTURE.md`](./02-ARCHITECTURE.md)
3. Confirm your change fits a layer boundary
4. If architecture changes, add/update an ADR in `docs/decisions/`

## PR/change checklist

- [ ] Change is scoped to one clear concern
- [ ] Module boundaries remain clean
- [ ] Public APIs are documented with intent
- [ ] Tests included (or rationale documented)
- [ ] Docs updated for behavioral changes

## Documentation style

- Prefer short sections and bullets
- Define terms before using abbreviations
- Keep examples concrete
- Assume zero prior project context

## AI collaboration contract

If you are an AI contributor:

- State assumptions clearly
- Avoid broad rewrites without approval
- Prefer additive, reviewable commits
- Leave handoff notes in plain language
