## Tier 2 — High-value backlog (queued)

These were surfaced over the AFK arc but parked while we focused on UI consolidation + game systems.

### First-boot vision
**What**: Wire the renderer to actually display the spawn scene — mothership crew quarters with Earth visible through the window, fleet ships outside, docking activity.
**Status**: Data exists in `data/ships/starter_fleet.ron`, AI Perception API can read it, but renderer doesn't paint it for human players. The 3D world is currently empty void with one icosphere.
**Why deferred**: User said "no point until we have a working game" (this round). Pick up after Tier 1A is done.
**Effort**: Multi-session graphics work — wgpu pipeline integration with ship interiors, planet rendering, multiple-ship instances, window viewport, particle effects for engine plumes.

### AI sample client
**What**: A working Node.js script demonstrating the AI Perception API end-to-end against the live VPS.
**Status**: API shipped v0.131.0, never end-to-end-tested in production (registration auth issue blocked initial test, then pivoted away).
**Why valuable**: Validates v0.131.0 actually works in production; becomes the working example AI builders need.
**Effort**: ~200 LOC + a published doc page.

### Real backend admin endpoints
**What**: Dedicated `/api/v2/admin/*` REST surface for server admin operations.
**Status**: Currently routes through chat slash commands (kick/ban/lockdown/etc.) which works fine. A REST surface would be cleaner for non-chat tooling (CLI admin, scripts, web admin dashboard).
**Why deferred**: The slash-command path works; this is a refactor not a feature.
**Effort**: Medium — ~6 endpoints with proper auth.

### Server-side world `tick()`
**What**: Currently `GameWorld::tick()` is a no-op. World doesn't simulate when no humans are online.
**Status**: Stub exists in `src/relay/handlers/game_state.rs:444`. Needs NPC behaviors, time-of-day progression, weather simulation on the server.
**Why valuable**: AI agents perceive a frozen scene. Real worlds change.
**Effort**: Medium-large. Some overlap with Tier 1A — many client-side systems will have server-side counterparts.

### Game state persistence
**What**: `GameWorld` lives in memory only. Relay restart resets player movements + interactions.
**Status**: Not implemented. Need save/load to SQLite.
**Effort**: Small-medium.

### Perception API rate limiting
**What**: `game_perceive` has no rate limit. AI could spam.
**Effort**: Small. Add Fibonacci backoff like chat has.

### Multi-AI write contention
**What**: If 5 AIs `game_interact` on the same locker simultaneously — no locking, no transactional ordering.
**Effort**: Medium. Optimistic locking on entity components.

### Distribution sovereignty steps 5–7 (from previous roadmap)
- **Step 5**: Internet Archive seeder (~30 min)
- **Step 6**: Software Heritage harvest (~10 min)
- **Step 7**: WinGet manifest (~1 hour PR)
- **Step 2 (deferred)**: Codeberg mirror (operator awaiting signup email)

## Tier 3 — Lower priority but tracked

### Operational observability
**What**: Live VPS doesn't expose application tracing logs (only systemd lifecycle). When a bug ships, no monitoring catches it.
**Effort**: Medium — wire tracing to file output, add a `/api/v2/admin/logs` endpoint or similar.

### Self-asserted `subject_class_v1` is gameable
**What**: Humans can declare themselves AI to dodge governance voting (or vice versa). No proof-of-AI-ness mechanism.
**Status**: Acknowledged in design but no enforcement. May be acceptable as honor-system + audit trail.
**Effort**: Open design question, not just implementation.

### Backup verification
**What**: We *assume* the BitTorrent seeder, self-hosted release mirror, and layered packages cascaded correctly. Nobody actually downloads via each backup channel and verifies SHA.
**Effort**: Small CI job.

### Native chat parity (extended)
- Right-click context menus
- Drag-drop file upload
- Notification badge integration with OS taskbar
- Spell-check (egui has primitives)
