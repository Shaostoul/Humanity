## Tier 1 — Current focus (operator-prioritized 2026-05-01)

### A. Game Systems — finish the 16 stubs so the game is playable

**Where they are**: 16 systems are 47–52 LOC scaffolds that load data from RON but have empty `tick()` bodies. Players can move + perceive + interact with objects, but the world doesn't *do* anything when you're in it. Need to wire each system to actually mutate ECS components.

**Stub systems (16)**: aging, astronomy, creative_arts, docking, fire, genetics, geology, governance, hvac, manufacturing, medical, oceanography, offline, plumbing, transportation, waste

**Partial systems (2)**: electrical (120 LOC scaffold), psychology (144 LOC scaffold)

## Tier 2 — High-value backlog (queued)

**What**: Wire the renderer to actually display the spawn scene — mothership crew quarters with Earth visible through the window, fleet ships outside, docking activity.
**Status**: Data exists in `data/ships/starter_fleet.ron`, AI Perception API can read it, but renderer doesn't paint it for human players. The 3D world is currently empty void with one icosphere.

## Tier 2 (operator-flagged 2026-05-02 sprint)

### Async QA testing loop (shipped v0.159.0)
**What**: A Testing page where the developer (Claude) lists features to verify. Operator opens the page, reads each task (feature / what to test / expected), clicks Mark Passed or Report Issue (with a note). Result is posted to the active chat channel as a `[QA]`-tagged message — the developer reads it next session.
**Status**: ✅ shipped. Data file at `data/testing/qa_tasks.json` (infinite-of-X compliant). Page at `src/gui/pages/testing.rs`. Pre-loaded with 17 tasks covering chat parity, server settings, game systems, perception API, and the nav-redesign proposal.

### Nav redesign — Reality / Sim / Tools / Settings two-tier
**What**: Replace the current single-row nav (overwhelming, mixes real-world and game-world tools) with a two-tier structure. Top tier: ~4-5 wide categories (~64px tall, full screen wide). Sub tier: pages within the active category. Real/Sim toggle goes away — instead, Reality and Sim are separate top categories with their own Wallet/Market/etc, eliminating accidental real-money mistakes.
**Status**: Proposal queued as a Testing task (`design-nav-redesign`) for operator review. Implementation deferred until operator confirms the category split + sub-page assignments.

### In-app browser with isolated profiles + encrypted vault
**What**: Render web pages inside HumanityOS so users can log into external sites for account creation, OAuth, etc. Each site runs in an isolated browser profile so Facebook can't track Etsy can't track Amazon, etc. Passwords + cookies stored in the existing vault (AES-GCM + PBKDF2). Tasks like "create your account here" can hyperlink directly into this browser.
**Status**: Not implemented. Large arc. Likely needs CEF (Chromium Embedded Framework) or wry/tauri-style webview integration. `docs/engine/in_game_browser.md` already exists with prior design notes.
**Effort**: Multi-session graphics + IPC work.

### Player feedback / rewards system
**What**: Ability to recruit testers ("100 people to test this quest chain") and reward them. Reward mechanism TBD — could be reputation points, in-game currency, NFT badges, real crypto via the Solana wallet.
**Status**: Not implemented. Depends on in-app browser (account creation flow), QA testing loop (already built), and a proper rewards economy design.

### Custom window chrome / title bar
**What**: Replace the default OS title bar with a custom one styled with HumanityOS theme tokens. More polished, integrated with the nav.
**Status**: Not implemented. Small-medium egui work — toggle decorations off + draw our own min/max/close buttons + drag handle.

### Multi-AI write contention
**What**: If 5 AIs `game_interact` on the same locker simultaneously — no locking, no transactional ordering.
**Effort**: Medium. Optimistic locking on entity components.

### Distribution sovereignty steps 5–7 (from previous roadmap)
- **Step 5**: Internet Archive seeder (~30 min)
- **Step 6**: Software Heritage harvest (~10 min)
- **Step 7**: WinGet manifest (~1 hour PR)
- **Step 2 (deferred)**: Codeberg mirror (operator awaiting signup email)

## Tier 3 — Lower priority but tracked

### Operational observability
**What**: Live VPS doesn't expose application tracing logs (only systemd lifecycle). When a bug ships, no monitoring catches it.
**Effort**: Medium — wire tracing to file output, add a `/api/v2/admin/logs` endpoint or similar.

### Self-asserted `subject_class_v1` is gameable
**What**: Humans can declare themselves AI to dodge governance voting (or vice versa). No proof-of-AI-ness mechanism.
**Status**: Acknowledged in design but no enforcement. May be acceptable as honor-system + audit trail.
**Effort**: Open design question, not just implementation.

### Backup verification
**What**: We *assume* the BitTorrent seeder, self-hosted release mirror, and layered packages cascaded correctly. Nobody actually downloads via each backup channel and verifies SHA.
**Effort**: Small CI job.

### Native chat parity (extended)
- Right-click context menus
- Drag-drop file upload
- Notification badge integration with OS taskbar
- Spell-check (egui has primitives)
